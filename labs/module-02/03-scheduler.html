Concept · kube-scheduler

Where should this pod run?

A new pod starts with no node assigned. The scheduler decides in two phases: filter out nodes that can't host it, then score the survivors and pick the best. Configure the pod and watch both phases.

The pod to schedule — cdr-processor

The operator's CDR (Call Detail Record) processor. Set its requirements; the scheduler re-evaluates instantly.

Phase 1 & 2 — filter, then score

Red = filtered out (can't host the pod). Blue = feasible. Green = the winner (highest score).

What the scheduler weighs

Filter (predicates) — can it run here?Enough free CPU/memory · required node labels (nodeSelector/affinity) · taints the pod doesn't tolerate · port conflicts
Score (priorities) — which feasible node is best?Spread pods of the same app across nodes · prefer more free capacity · image already cached · data locality
The scheduler only chooses — it writes a pod→node binding through the API server. The chosen node's kubelet notices via its watch and actually runs the pod (Visualization 04).
Taints & tolerations work in reverse: a taint repels pods from a node unless the pod carries a matching toleration. Here, node-3 is tainted gpu=true:NoSchedule, so it's filtered out until you add the toleration.
