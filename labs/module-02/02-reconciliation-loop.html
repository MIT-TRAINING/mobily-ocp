The loop

observe → compare → act → repeat. The controller watches actual pods, compares to your desired count, and creates or deletes pods to close the gap. Forever.
Turn off “Controller running” to simulate a broken control plane — now killed pods stay dead. That’s what the controllers protect you from.

Where this self-healing comes from

ReplicaSet controlleractual pod count drifted from desired → create/delete pods
Node controllernode went NotReady → evict its pods so they reschedule
Schedulereach replacement pod has no node → gets placed (Viz 03)
You changed spec.replicas; the system changed status. You own desired state; Kubernetes owns actual state. This same loop is also why GitOps works — keep desired state in Git and a controller reconciles the cluster to match.
