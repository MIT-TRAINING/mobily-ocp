The two golden rules

1 · Everything goes through the API server No component edits etcd or another component directly. You, the scheduler, the controllers, and every kubelet all talk only to kube-apiserver. One front door = one place for auth, validation, and audit.
2 · etcd is the only place state lives Every other component is effectively stateless — it can crash and rebuild its view by reading from the API server. Back up etcd and you back up the whole cluster.
In production (and OpenShift by default) the control plane runs on 3 nodes. etcd uses Raft consensus and needs a quorum (majority) — which is why you want an odd number. Lose 1 of 3 and the cluster keeps working.
