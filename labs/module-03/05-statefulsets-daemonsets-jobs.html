Concept · The special controllers

When a Deployment isn't the right shape.

Three controllers cover what a Deployment can't: StatefulSets (stable identity + own disk), DaemonSets (one pod per node), and Jobs/CronJobs (run to completion). Explore each below.

StatefulSet cdr-store

Pods are created in order, get stable ordinal names, and each keeps its own PVC. Scale and watch identities stay put.

3
cdr-store-0 is always -0, bound to data-cdr-store-0 forever. Delete & reschedule it and it returns with the same name and the same disk. Pair with a headless Service for per-pod DNS cdr-store-0.cdr-store.

DaemonSet node-probe

One pod per eligible node — you never set a count. Add nodes and watch pods appear. The control-plane is tainted, so no probe lands there.

DESIRED: 2 · CURRENT: 2
"One per node" means one per node the pod is allowed on. The control-plane carries NoSchedule and our DaemonSet has no toleration — so it stays empty. System DaemonSets (CNI) add tolerations to cover it.

Job cdr-roundup

completions: 3, parallelism: 2 — run 3 successes, two at a time, then stop. Pods stay Completed (not restarted) so you can read logs.

STATUS: ready · COMPLETIONS: 0/3
A Job's restartPolicy is Never/OnFailure — never Always. A CronJob just creates a Job like this on a schedule (e.g. 0 2 * * * = nightly billing).
