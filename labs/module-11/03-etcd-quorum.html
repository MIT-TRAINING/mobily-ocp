Concept · etcd Administration

Lose a minority, survive. Lose the majority, lose the cluster.

etcd is the cluster's brain: 3 members on the control plane, kept consistent by Raft. Writes need a majority (quorum). Click members to fail them and watch the API stay up — until quorum breaks. Then push fsync latency up and watch a "healthy" cluster still go sick.

3-member control plane — click a member to fail/recover it

3 of 3 up quorum: 2 needed
✅ etcd has quorum — API server accepts writes
Quorum = majority = floor(N/2)+1. With 3 members you need 2. Fail one → still 2, healthy. Fail a second → only 1 (a minority) → quorum lost, the API goes read-only. This is why control planes have an odd number of masters.

Healthy isn't only "up" — it's also "fast"

p99 fsync: 3 ms
etcd commits nothing until it fsyncs the write-ahead log to disk — so disk latency is etcd latency. Below ~10 ms all members are up and stable. Drag it up: slow disk causes leader elections, then API timeouts — a cluster that's "all members up" yet unwell.

How you'd actually check it

oc rsh -n openshift-etcd etcd-master-0
etcdctl endpoint health --cluster   # each member: healthy + latency
etcdctl endpoint status -w table    # leader flag, DB size, raft term
Prometheus watches etcd_disk_wal_fsync_duration_seconds, etcd_server_leader_changes_seen_total, and etcd_mvcc_db_total_size_in_bytes; platform alerts like etcdHighFsyncDurations and etcdMembersDown page you before it's fatal. Backup/restore is Module 12.
