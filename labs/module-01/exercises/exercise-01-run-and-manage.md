# Exercise 1 — Run & Manage Containers

> **Scenario:** The **Number Portability (MNP)** team needs a quick status
> endpoint to check whether a ported number's migration is complete. You'll run it
> as a container and manage its lifecycle.

| | |
|---|---|
| **Practice** | `run`, `ps`, `logs`, `exec`, `stop/start`, `rm` |
| **Maps to** | Visualization 04 (Lifecycle), Demo 1 |
| **Difficulty** | ⭐ Beginner · ~20 min |

---

## Tasks

1. **Run** a detached container named `mnp-status` from
   `registry.access.redhat.com/ubi9/httpd-24:latest`, publishing host port
   **8080** to container port **8080**, with the command `httpd -DFOREGROUND`.
   > Hint: the httpd-24 image serves files from `/var/www/html` on port 8080.

2. **Verify** it is running and note its `STATUS` and `PORTS`.

3. Using `exec`, create a status document inside the container at
   `/var/www/html/port-status.json` containing:
   ```json
   {"msisdn":"966500000007","port_request":"PR-10293","status":"COMPLETED"}
   ```

4. From the host, **retrieve** that document over HTTP and confirm the JSON.

5. Inspect the container's **CPU and memory** usage (a one-shot reading).

6. Drive the **lifecycle**: stop the container, confirm it shows `Exited`, then
   start it again and confirm the status document **survived** the restart.

7. Now **remove** the container (force). Explain (in a comment to yourself) what
   happened to `port-status.json`.

---

## Validation

After Task 4:
```bash
curl -s http://localhost:8080/port-status.json
# Expected:
# {"msisdn":"966500000007","port_request":"PR-10293","status":"COMPLETED"}
```

After Task 6 (restart):
```bash
podman ps --format "table {{.Names}}\t{{.Status}}"
curl -s http://localhost:8080/port-status.json   # still returns the JSON
```

After Task 7:
```bash
podman ps -a | grep mnp-status || echo "Removed — and so is its writable layer"
```

## Stretch goals

- Run a **second** instance `mnp-status-2` on port `8081` from the **same image**.
  How many image copies are on disk? (`podman images`)
- Use `podman logs` to view the HTTP access log entry generated by your `curl`.
- Restart-policy: re-run with `--restart=on-failure` and reason about when it
  would help.

## Reflection

- Where did `port-status.json` physically live?
- Why did it survive `stop`/`start` but not `rm`?
- What single command would you use to debug a running container without stopping
  it?

<details>
<summary><strong>✅ Solution</strong> (try the tasks first)</summary>

```bash
# 1. Run
podman run -d --name mnp-status -p 8080:8080 \
  registry.access.redhat.com/ubi9/httpd-24:latest httpd -DFOREGROUND

# 2. Verify
podman ps --format "table {{.Names}}\t{{.Status}}\t{{.Ports}}"

# 3. Inject the status document
podman exec mnp-status bash -c \
  'echo "{\"msisdn\":\"966500000007\",\"port_request\":\"PR-10293\",\"status\":\"COMPLETED\"}" > /var/www/html/port-status.json'

# 4. Retrieve
curl -s http://localhost:8080/port-status.json

# 5. Resource usage (cgroups)
podman stats --no-stream mnp-status

# 6. Lifecycle
podman stop mnp-status
podman ps -a --format "table {{.Names}}\t{{.Status}}"   # Exited (0)
podman start mnp-status
curl -s http://localhost:8080/port-status.json          # survived (writable layer kept)

# 7. Remove
podman rm -f mnp-status
# port-status.json lived in the container's WRITABLE LAYER, which rm destroys.
# Stop/start keeps the writable layer; rm deletes it permanently.

# Stretch
podman run -d --name mnp-status-2 -p 8081:8080 \
  registry.access.redhat.com/ubi9/httpd-24:latest httpd -DFOREGROUND
podman images   # one image, shared by both containers
podman logs mnp-status-2
podman rm -f mnp-status-2
```

**Key point:** one image → many containers; the writable layer is per-container
and is destroyed by `rm`. Durable data needs a volume (Exercise 4).
</details>
