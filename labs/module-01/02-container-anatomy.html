Concept · Container architecture & Linux primitives

Build a container from a plain process

A container is just a Linux process with two things added: namespaces (what it can see) and cgroups (what it can use). Toggle them on and watch an ordinary process turn into a container.

① Namespaces — isolation (what the process can see)

Turn each namespace on to give the process its own private view of that resource. With none on, it sees the whole host.

Status

A process becomes a true container once it has its own filesystem (MNT), process tree (PID), and network (NET) views — the rest harden it further.

② cgroups — resource limits (what the process can use)

Namespaces hide things; cgroups cap things. Set a memory limit, then push the workload's demand past it.

This is exactly how Kubernetes/OpenShift requests & limits work: exceed the memory limit and the kernel OOMKills the container.

Putting it together

NamespacesIsolation — own view of PID, NET, MNT, UTS, IPC, USER
cgroupsLimits — cap CPU, memory, I/O
Capabilities / seccomp / SELinuxConfinement — what it may do
All three are kernel features. There is no separate "container engine magic" — the runtime (CRI-O, containerd) just asks the kernel to apply them to a process.
