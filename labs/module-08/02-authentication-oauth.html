Concept · Authentication

OpenShift doesn't store your password. It asks an identity provider.

The OAuth server handles logins but delegates the credential check to a configured identity provider. Follow a login end-to-end, compare IdP types, then wire up an htpasswd provider.

The login flow

Step through what happens when a Mobily engineer runs oc login.

🔑 oc login -u m.hassan — request hits the cluster
▼
🛡️ OAuth server — doesn't check the password itself; routes to an IdP
▼
🗂️ Identity Provider — verifies credentials (htpasswd / LDAP / OIDC)
▼
🎫 Token issued — User + Identity objects created on first login
OpenShift has no built-in human user database. The OAuth server (the authentication Cluster Operator) delegates to whatever IdPs you configure in the OAuth resource.

Wire up htpasswd (the lab/break-glass IdP)

1 · Build the file (bcrypt): htpasswd -c -B -b users.htpasswd alice '<pw>'
2 · Store as a Secret: oc create secret generic mobily-htpasswd --from-file=htpasswd=users.htpasswd -n openshift-config
3 · Add an htpasswd entry to oc edit oauth cluster (references the Secret)
4 · Wait for the OAuth Operator to roll out, then oc login -u alice
5 · Grant access — oc adm policy add-role-to-user edit alice -n team-billing (authn ≠ authz!)
A brand-new user can log in but can do nothing until a Role is bound. And keep one htpasswd admin as break-glass even after wiring LDAP — then remove kubeadmin.
