Concept · Troubleshooting methodology

The status string is already a diagnosis.

Troubleshooting is narrowing by layer: answer where before why. A pod's status tells you the class of problem before you read a single log. Pick a symptom below and walk the same loop every time — status → events → logs → fix.

The loop, every time: 1 get (status = where) 2 describe/events (why) 3 logs [--previous] 4 smallest fix 5 confirm + must-gather

Where does it live — app or platform?

App layer (your namespace): oc get pods → describe → logs. Status strings like Pending / ImagePullBackOff / CrashLoopBackOff / 0/1 Ready.   Platform layer: oc get co, oc get nodes, oc get etcd, oc adm node-logs. Degraded operators, NotReady nodes, pending CSRs.
Events narrate failures. oc get events --sort-by=.lastTimestamp is the golden first move — OpenShift writes FailedScheduling / ImagePullBackOff / Unhealthy / FailedMount there before you ever open a log. Events = where; logs = why.
