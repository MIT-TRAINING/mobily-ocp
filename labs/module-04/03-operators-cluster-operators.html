Concept · Software that operates software

OpenShift runs itself with Operators.

An Operator is a controller that encodes a human expert's ops knowledge as a reconcile loop. First drive one yourself; then see how OpenShift's own platform — ~30 Cluster Operators coordinated by the CVO — is built the exact same way.

An Operator's reconcile loop

A Mobily team installs an SMSGateway Operator. You edit one Custom Resource — kind: SMSGateway, size: N — and the Operator makes reality match. Scale it, then break it.

watch CR→ diff desired vs actual→ act↺
3
desired: 3 actual: 3 reconciled ✓

Resources the Operator manages

You edit the Custom Resource (desired state); the Operator reconciles the real Deployments, Services, Secrets and Routes to match. This is the Module 2 reconcile loop — now applied to a whole application, not just pods.

The CRD + CR that make it possible

CRD — extends the API

Installed once (by OLM). After this, SMSGateway is as real to the API server as Deployment.

kind: CustomResourceDefinition
name: smsgateways.telecom.mobily
→ unlocks oc get smsgateway, RBAC, etcd storage — free.

CR — an instance (desired state)

kind: SMSGateway
metadata: { name: sms-prod }
spec:
  size: 3
  tlsEnabled: true

The Operator watches CRs of this kind and reconciles.

OpenShift's own platform: Cluster Operators

The same pattern runs OpenShift itself. oc get clusteroperators shows ~30 of them — each owns one slice of the platform, each reports Available / Progressing / Degraded. Trigger an upgrade, then break one and triage it.

clusterversion: 4.18.3 · healthy
The Cluster Version Operator (CVO) reads the desired release image and drives every Cluster Operator to that version in dependency order. When triaging a sick cluster, the first column to scan is DEGRADED.
